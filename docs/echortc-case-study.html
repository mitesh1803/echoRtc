01 — Overview

What is EchoRTC?

EchoRTC is a real-time video conferencing application I built to understand how platforms like Zoom and Google Meet actually work under the hood — not by using a hosted SDK, but by implementing the full media routing stack from scratch.

The project covers the complete lifecycle of a video call: peer discovery, WebRTC transport negotiation, SFU media routing, in-call chat, screen sharing, browser-based speech recognition, and AI-generated meeting summaries via Gemini.

02 — Problem Space

Why build this at all?

The learning gap

Most developers who "build a video app" wire up Twilio Video or Daily.co in an afternoon. They learn the API surface of a vendor, not the underlying technology. When asked "how does WebRTC work?" in an interview, they can't answer.

I wanted to be the person who could answer it — and explain why Zoom doesn't use peer-to-peer networking.

The technical question

The central question was: how do you route audio and video between N participants efficiently?

The naive answer (direct peer connections) breaks at 3–4 people. Understanding why, and how to solve it, is what this project is really about.

"Most video apps abstract away the hard parts. I wanted to understand the hard parts."

Secondary goals: get comfortable with complex async flows (WebSocket signaling, mediasoup transport lifecycle), build something with genuine AI utility beyond a chatbot, and produce a project that demonstrates full-stack depth — not just "built a React app."

03 — Architecture

SFU over mesh — the core decision

A naive multi-party video call uses a full mesh: every peer connects directly to every other peer. With N participants, each peer maintains N-1 connections and uploads their stream N-1 times. At 6 people, each client is uploading 5 streams simultaneously. This collapses quickly on real-world upload bandwidth.

// Peer-to-peer mesh — N² connections Peer A ←────────────→ Peer B ↕ ↕ Peer C ←────────────→ Peer D // At 4 peers: 6 connections, each peer uploads 3× streams // At 6 peers: 15 connections — collapses on mobile / weak upload // EchoRTC — SFU (Selective Forwarding Unit) Peer A ──upload──→ mediasoup ──forward──→ Peer B Peer B ──upload──→ server ──forward──→ Peer C Peer C ──upload──→ ──forward──→ Peer A // Each peer uploads once, server routes to all consumers // No decoding/re-encoding — raw RTP packet forwarding only

An SFU (Selective Forwarding Unit) puts a server in the middle. Each peer uploads their stream once to the server; the server forwards RTP packets to each subscriber independently. Crucially, the server does not decode or re-encode anything — it forwards raw packets, keeping CPU cost minimal.

Backend components

mediasoup worker — native C++ process handling RTP routing. One per CPU core in production.

mediasoup router — one per room, holds codec config (VP8 + Opus).

WebRTC transports — one send + one recv per peer, handling ICE/DTLS negotiation.

WebSocket signaling — raw ws library, 7-event protocol handling room lifecycle.

Frontend components

mediasoup-client Device — loaded with server's RTP capabilities, creates transports client-side.

Producers — one per local track (camera, mic, screen). Each produces to the send transport.

Consumers — created per remote producer. Each renders a remote track.

Speech recognition — Web Speech API, continuous mode, chunks sent over WebSocket as transcript events.

// Room state (in-memory, per server instance) Room { router: mediasoup.Router // codec + RTP config peers: Map<peerId, Peer> // all connected clients producerPeerMap: Map<producerId, peerId> // reverse lookup transcript: TranscriptChunk[] // speech-to-text buffer } Peer { socket: WebSocket // signaling channel sendTransport: WebRtcTransport // client → server media recvTransport: WebRtcTransport // server → client media producers: Map<id, Producer> // tracks this peer sends consumers: Map<id, Consumer> // tracks this peer receives rtpCapabilities: RtpCapabilities // what this browser can decode }
04 — Signaling Flow

From connection to media — 7 steps

Every participant follows a deterministic 7-step handshake before any media flows. This sequence is the core of the mediasoup protocol.

01
joinRoom Client sends joinRoom with roomId and displayName. Server creates the room (if new) and the Peer object. Returns routerRtpCapabilities — what codecs the server supports.
02
Device load + rtpCapabilities exchange Client loads the mediasoup-client Device with server caps. Sends back its own rtpCapabilities — what codecs this browser can decode. Server stores this per-peer to filter consumers correctly.
03
createTransport (send) Server creates a WebRtcTransport on the mediasoup router. Returns ICE parameters, ICE candidates, and DTLS parameters to the client. Client creates its local send transport.
04
connectTransport + produce Client fires connectTransport with DTLS parameters (completing the DTLS handshake). Then produces local tracks. Each produce triggers a correlationId-based ack to match concurrent produce calls.
05
newProducer broadcast Server notifies all other peers in the room that a new producer exists. Each peer sends a consume request back. Server checks router.canConsume() before creating the consumer.
06
createTransport (recv) + consumed Recv transport is created after send is established. Server creates a Consumer object and returns its RTP parameters. Client receives the track and renders it in a new MediaStream.
07
existingProducers for late joiners When a peer joins a room that already has participants, the server sends an existingProducers array on recv transport creation. The client iterates and consumes each one — no race conditions with live joins.
05 — Key Decisions

Why these choices?

Decision Chosen Considered & rejected Reasoning
Media server mediasoup Twilio Video, Daily.co, Agora Hosted SDKs abstract the protocol. mediasoup is raw SFU — forces understanding of ICE, DTLS, RTP routing at the packet level.
Signaling transport Raw WebSocket (ws) socket.io socket.io adds reconnect, rooms, namespaces — abstractions I wanted to implement myself to understand the tradeoffs. WebSocket keeps the contract explicit.
AI integration Gemini 1.5 Flash OpenAI GPT-4o Cost — Gemini Flash has a generous free tier, making the feature deployable without a paid API plan. Structured JSON output prompt works equally well.
Frontend framework React + Vite + TypeScript Next.js No SSR needed — the entire app is client-side. Vite's HMR keeps iteration fast. TypeScript catches the complex mediasoup type interactions at compile time.
Screen share layout Presentation mode (pinned + sidebar) Treat screen as another grid tile A screen share tile in a grid becomes too small to read. Pinning it as the main view with participants in a sidebar matches how Zoom/Meet handle it.
Transcript delivery Web Speech API (client-side) Whisper / Deepgram (server-side) Client-side avoids sending raw audio to a server, reducing bandwidth and latency. The trade-off is English-only support and browser dependency.

07 — Technical Challenges

Hard problems solved

Challenge 01

Concurrent produce calls creating race conditions

The mediasoup client fires a produce event for each track. Both audio and video produce simultaneously, both sending produce messages to the server. The server responds with produced + a producerId. Without tracking which response belongs to which produce call, the audio producer could get the video ID and vice versa.

Used a correlationId (random UUID) injected per produce call. Server echoes it back in the produced response. Client event listener filters on correlationId match before resolving the callback. No race condition possible.
Challenge 02

Late joiners missing existing producers

When a peer joins a room that already has participants, those participants' newProducer events have already been broadcast and are gone. The late joiner had no way to discover existing streams.

On recv transport creation, the server scans the room's peer map and sends an existingProducers array containing all currently active producers. The client handles this identically to live newProducer events.
Challenge 03

Screen share and camera on the same peer

A peer sharing their screen produces three tracks: camera video, microphone audio, and screen video. The consumer side needed to know which video track was the camera and which was the screen — to render them in the right UI slots.

Added appData: { mediaTag: 'screen-video' } to the screen producer. Server forwards mediaTag in both newProducer and consumed messages. Client uses a peerId:screen namespace key in the streams map to separate the two video streams per peer.
Challenge 04

React re-renders not picking up new tracks on existing streams

When a new consumer track arrived for a peer that already had a stream, adding the track to the existing MediaStream object didn't trigger a React re-render — same object reference, no state change detected.

On every new consumer track, always construct a brand-new MediaStream with all previous tracks plus the new one. The new object reference guarantees React detects the state change and the RemoteVideo component's useEffect re-fires to re-attach srcObject.
Challenge 05

Peer disconnect leaving orphaned consumers on other clients

When a peer disconnected, their producers were closed on the server. But other peers still had consumer objects pointing to those producers. Their video tiles stayed frozen rather than disappearing.

The disconnect handler collects the leaving peer's producer IDs before cleanup. It then iterates all other peers, closing any consumer whose producerId is in that set, and sends a peerLeft event. The client removes that peerId from its remoteStreams map.
08 — What I Learned

Key takeaways

  • The difference between mesh, SFU, and MCU architectures is not academic. It has direct consequences for bandwidth, latency, and server cost. Understanding this lets you make informed architecture decisions in any real-time system, not just video.
  • ICE, DTLS, and RTP are separate protocols that compose to form a WebRTC connection. ICE finds a network path, DTLS establishes encrypted transport, RTP carries the media. Each can fail independently — knowing which layer failed makes debugging possible.
  • Async ordering in Node.js is deceptive. Two awaits that look sequential can produce interleaved results when both are triggered by external events. The correlationId pattern is a general solution to this class of problem, not just a WebRTC trick.
  • React state and DOM refs are genuinely different mental models. srcObject on a video element is a DOM property, not a React prop — it must be set imperatively in a useEffect. Mixing declarative and imperative paradigms cleanly requires thinking about which model owns which piece of state.
  • AI features are most useful when they solve a real problem in the flow, not bolted on as a demo. Meeting transcription and summarisation has genuine value — people leave meetings without clear action items. Building it as part of the call lifecycle (not a post-processing step) taught me to think about AI as infrastructure, not a feature flag.
  • Deployment configuration is part of the product. Having a Dockerfile, docker-compose, and nixpacks config isn't DevOps — it's the difference between a project that only works on your machine and one that can be evaluated by anyone.
